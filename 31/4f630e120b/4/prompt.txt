# Plan: Add Real-Time Streaming to CLI and Provider

## Context

The `arcan run` CLI command (implemented in the previous session) works but is silent during LLM calls — the CLI sends POST /runs, waits for completion, then fetches events. For slow models like Ollama, this means 30+ seconds of no output (and HTTP timeouts). The user wants:

1. **CLI SSE streaming**: Display events (tool calls, text, errors) as they arrive in real-time
2. **Provider streaming**: Send `"stream": true` to Ollama/OpenAI to avoid timeouts and enable per-token text display

## Architecture Overview

Current event flow: `Provider.complete()` → `ArcanProviderAdapter` (spawn_blocking) → `KernelRuntime.tick_on_branch()` → `EventStore` (persist) → `broadcast::Sender<EventRecord>` → SSE endpoint → client.

The Provider trait is synchronous, returns atomic `ModelTurn`. The broadcast channel already supports live event streaming to SSE clients. The TUI uses `reqwest-eventsource` for SSE consumption (pattern to reuse).

## Changes

### 1. CLI SSE Streaming — `arcan/crates/arcan/src/cli_run.rs`

Rewrite `run_cli()` from poll-after-completion to concurrent SSE streaming:

1. Get current event cursor via `GET /sessions/{id}/events` (latest sequence)
2. Open SSE stream at `/sessions/{id}/events/stream?cursor={cursor}&format=canonical`
3. Spawn POST `/sessions/{id}/runs` concurrently (don't await before reading SSE)
4. Read SSE events from the stream, render to stdout as they arrive:
   - `TextDelta` / `AssistantTextDelta`: write delta text (skip if sequence=0 already showed it)
   - `ToolCallRequested`: `[tool: name] requested`
   - `ToolCallCompleted`: `[tool: name] OK`
   - `RunFinished`: newline + close stream, exit 0
   - `RunErrored`: print error + exit 1
5. Track `saw_streaming_text` flag — if ephemeral TextDelta events (sequence 0) arrived, skip persisted ones (sequence > 0) to avoid duplicate output
6. `--json` flag: output raw SSE data payloads instead of formatted text

**Dependencies to add**: `reqwest-eventsource = "0.6"`, `futures-util.workspace = true` (to `arcan/crates/arcan/Cargo.toml`)

### 2. Provider Trait Extension — `arcan/crates/arcan-core/src/runtime.rs`

Add optional streaming method to `Provider` trait (default falls back to `complete()`):

```rust
fn supports_streaming(&self) -> bool { false }

fn complete_streaming(
    &self,
    request: &ProviderRequest,
    on_text: &dyn Fn(&str),
) -> Result<ModelTurn, CoreError> {
    self.complete(request)
}
```

Both methods have default implementations, so existing providers (Anthropic, Mock) are unaffected.

### 3. OpenAI Provider Streaming — `arcan/crates/arcan-provider/src/openai.rs`

Add streaming support to `OpenAiCompatibleProvider`:

- Add `pub enable_streaming: bool` to `OpenAiConfig` (default `true` for Ollama, `false` for OpenAI)
- Override `supports_streaming()` → returns `self.config.enable_streaming`
- Override `complete_streaming()`:
  1. Add `"stream": true` to request body
  2. Send request (using non-streaming reqwest to get the response, then read body as a buffered reader line-by-line)
  3. Parse each `data: {...}` line as an SSE chunk:
     - `choices[0].delta.content` → call `on_text(content)`, accumulate into full text
     - `choices[0].delta.tool_calls` → accumulate tool call arguments
     - `choices[0].finish_reason` → set stop reason
     - `data: [DONE]` → break
  4. Build and return `ModelTurn` from accumulated text + tool calls

**Streaming response types** (add to the module):
```rust
#[derive(Debug, Deserialize)]
struct StreamChunk {
    choices: Vec<StreamChoice>,
    #[serde(default)]
    usage: Option<ApiUsage>,
}

#[derive(Debug, Deserialize)]
struct StreamChoice {
    delta: StreamDelta,
    finish_reason: Option<String>,
}

#[derive(Debug, Deserialize)]
struct StreamDelta {
    #[serde(default)]
    content: Option<String>,
    #[serde(default)]
    tool_calls: Option<Vec<StreamToolCall>>,
}

#[derive(Debug, Deserialize)]
struct StreamToolCall {
    index: usize,
    #[serde(default)]
    id: Option<String>,
    #[serde(default)]
    function: Option<StreamToolCallFunction>,
}

#[derive(Debug, Deserialize)]
struct StreamToolCallFunction {
    #[serde(default)]
    name: Option<String>,
    #[serde(default)]
    arguments: Option<String>,
}
```

### 4. Adapter Streaming Bridge — `arcan/crates/arcan-aios-adapters/src/provider.rs`

Wire provider streaming to the runtime's broadcast channel for real-time SSE delivery:

- Add `streaming_sender: Arc<std::sync::Mutex<Option<broadcast::Sender<EventRecord>>>>` field
- Update `ArcanProviderAdapter::new()` to accept the shared sender
- In `complete()`:
  1. Check `provider.supports_streaming()` AND streaming_sender is available
  2. If streaming: inside `spawn_blocking`, call `provider.complete_streaming()` with a callback that does `broadcast_sender.send(ephemeral_event)` where `ephemeral_event` has `sequence: 0`
  3. If not streaming: fall back to existing `provider.complete()`

The callback constructs EventRecord with the session_id from the request and `sequence: 0` (marking it as ephemeral/non-persisted). `broadcast::Sender::send()` is synchronous so it works inside `spawn_blocking`.

**Dependency**: Add `aios-protocol.workspace = true` to `arcan-aios-adapters/Cargo.toml` (for `EventRecord`, `EventKind`, `SessionId`, `BranchId`)

### 5. KernelRuntime Event Sender — `aiOS/crates/aios-runtime/src/lib.rs`

Add a public method to expose the broadcast sender:

```rust
pub fn event_sender(&self) -> broadcast::Sender<EventRecord> {
    self.stream.clone()
}
```

### 6. SSE Filter Fix — `arcan/crates/arcand/src/canonical.rs`

Allow ephemeral events (sequence 0) through the SSE stream filter. Change line ~367:

```rust
// Before:
&& event.sequence > cursor

// After:
&& (event.sequence > cursor || event.sequence == 0)
```

### 7. Wire Streaming Sender — `arcan/crates/arcan/src/main.rs`

After runtime creation, set the broadcast sender on the adapter:


Update `OpenAiConfig` construction to set `enable_streaming: true` for Ollama provider.
