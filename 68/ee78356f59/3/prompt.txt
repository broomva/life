 Phase 5: Connect Praxis FS Tools to Lago Event-Sourced Persistence

 Context

 Praxis FS tools (ReadFile, WriteFile, EditFile, ListDir, Glob, Grep)
 use raw std::fs calls. After every tool execution, a LakeFsObserver in
 arcan's main.rs does a post-hoc full-workspace scan to reconcile with
 Lago's event journal. This is O(workspace_size) per tool call —
 rebuilding the entire manifest from the journal, snapshotting every
 file, diffing, then emitting events. Wasteful and slow for large
 workspaces.

 Goal: Replace the post-hoc scan with inline write tracking. When Praxis
  writes a file, Lago knows immediately — O(1) per write instead of O(n)
  per tool call.

 Constraint: Praxis depends ONLY on aios-protocol (enforced by
 verify_dependencies.sh). Praxis CANNOT depend on Lago crates. The
 abstraction must live in praxis-core.

 Design

 Praxis FS Tools (ReadFile, WriteFile, EditFile, ListDir, Glob, Grep)
          │
          ▼
    praxis-core::FsPort (trait)
          │
     ┌────┴────┐
     ▼         ▼
  LocalFs    LagoTrackedFs
 (praxis)    (arcan-lago)
   std::fs    LocalFs + blob store + channel
                       │
                       ▼
               Background async task
               → Lago Journal append

 - FsPort: Sync trait in praxis-core abstracting filesystem ops with
 workspace boundary enforcement
 - LocalFs: Default implementation wrapping std::fs + FsPolicy (backward
  compatible)
 - LagoTrackedFs: Decorator in arcan-lago — delegates I/O to LocalFs,
 additionally hashes + stores blobs + sends write notifications via
 bounded mpsc channel
 - Background event writer: Async task receiving notifications and
 appending FileWrite events to Lago journal

 Steps

 1. Add FsPort trait + LocalFs to praxis-core

 NEW praxis/crates/praxis-core/src/fs_port.rs:

 pub trait FsPort: Send + Sync {
     fn workspace_root(&self) -> &Path;
     fn resolve(&self, path: &Path) -> PraxisResult<PathBuf>;
     fn resolve_for_write(&self, path: &Path) -> PraxisResult<PathBuf>;
     fn read_to_string(&self, path: &Path) -> PraxisResult<String>;
     fn read_bytes(&self, path: &Path) -> PraxisResult<Vec<u8>>;
     fn write(&self, path: &Path, content: &[u8]) -> PraxisResult<()>;
     fn exists(&self, path: &Path) -> bool;
     fn metadata(&self, path: &Path) -> PraxisResult<FsMetadata>;
     fn read_dir(&self, path: &Path) -> PraxisResult<Vec<FsDirEntry>>;
     fn create_dir_all(&self, path: &Path) -> PraxisResult<()>;
     fn relative(&self, absolute_path: &Path) -> Option<PathBuf>;
 }

 Supporting types: FsMetadata { is_file, is_dir, size_bytes },
 FsDirEntry { name, is_file, is_dir }.

 pub struct LagoTrackedFs {
     local_fs: LocalFs,

 Scope Boundaries

 In scope: ReadFileTool, WriteFileTool, EditFileTool, ListDirTool,
 GlobTool, GrepTool
 Out of scope: BashTool (SandboxPolicy), ReadMemoryTool/WriteMemoryTool
 (memory_dir), arcan-spaces tools, arcan-lago MemoryQueryTool

 Risk Assessment

 Risk: Channel full (missed events)
 Mitigation: Bounded 1000-item buffer + warn log; periodic fallback scan

   can be added later
 ────────────────────────────────────────
 Risk: BlobStore.put() contention
 Mitigation: redb handles concurrent readers; single writer per session
 ────────────────────────────────────────
 Risk: Test refactor breakage
 Mitigation: Mechanical change (FsPolicy → Arc LocalFs); same logic
 ────────────────────────────────────────
 Risk: Dependency cycle
 Mitigation: FsPort in praxis-core (not Lago); arcan-lago → praxis-core
   is allowed
 ────────────────────────────────────────
 Risk: Event loss on crash
 Mitigation: Acceptable: tracking is best-effort; manual reconciliation
   available via lago-fs snapshot
 ────────────────────────────────────────
 Risk: Performance regression
 Mitigation: Removes O(n) scan; adds O(1) hash+send per write — net
   improvement
